== Introduction

CYNA est une plateforme SaaS de cybersécurité complète, composée de trois services Dockerisés :

* **API Go** — Backend REST (port 8080)
* **Node.js** — Serveur web + proxy + interface admin (port 3000)
* **PostgreSQL** — Base de données (port 5432)

Ce guide vous accompagne pas à pas dans l'installation et la configuration complète de la plateforme.

=== Prérequis matériels

[cols="1,1", options="header"]
|===
| Composant | Minimum Recommandé
| **RAM** | 4 Go (8 Go recommandé)
| **CPU** | 2 cœurs (4 cœurs recommandé)
| **Espace disque** | 10 Go (20 Go avec backups)
| **OS** | Linux (Ubuntu 22.04+, Debian 12+, ou WSL2 sur Windows)
|===

== Étape 2 : Configuration

=== Créer le fichier `.env`

[source,bash]
----
cp .env.example .env
nano .env
----

=== Variables essentielles

[source,env]
----
# ── Base de données (ne pas modifier si utilisation Docker) ──
DB_HOST=db
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=password
DB_NAME=mydb

# ── Stripe (paiement en ligne — optionnel mais recommandé) ──
STRIPE_SECRET_KEY=sk_test_votre_cle_privee
STRIPE_PUBLISHABLE_KEY=pk_test_votre_cle_publique

# ── API Security ──
API_SECRET=une_chaine_aleatoire_tres_longue_32_caracteres_min

# ── Email (Resend) ──
RESEND_API_KEY=re_votre_cle_api_resend

# ── Sauvegardes automatiques (optionnel) ──
BACKUP_INTERVAL_MINUTES=1440  # Toutes les 24h
----

TIP: Pour générer une clé API_SECRET aléatoire : `openssl rand -hex 32`

== Étape 4 : Sauvegardes

=== Principe

Les sauvegardes utilisent `pg_dump` pour créer des fichiers `.sql` dans le volume Docker `backups_data`. Aucune configuration supplémentaire n'est nécessaire.

=== Types de sauvegarde

[cols="1,2,2", options="header"]
|===
| Type | Contenu | Usage
| `full` | Toutes les tables | Sauvegarde complète
| `data` | Toutes les tables sauf `api_logs` | Données métier
| `logs` | Uniquement `api_logs` | Audit et conformité
|===

=== Sauvegarde manuelle

. Ouvrir le panel admin : http://localhost:3000/backend/
. Aller dans **Sauvegardes**
. Cliquer sur **"Lancer la sauvegarde"**
. Le fichier `.sql` apparaît dans la liste

=== Sauvegarde automatique (planification)

==== Via l'interface

. Dans la section **Planification automatique**
. Choisir le type (Complète / Données / Logs)
. Définir l'intervalle (ex: `60` + `min` pour toutes les heures)
. Cliquer **"Ajouter"**

==== Via variable d'environnement

[source,env]
----
# Dans le fichier .env
BACKUP_INTERVAL_MINUTES=1440  # Toutes les 24h
----

Puis redémarrer :

[source,bash]
----
docker compose down && docker compose up -d
----

=== Restauration

. Dans la liste des sauvegardes, cliquer sur l'icône **restaurer** ↻
. Confirmer la restauration
. Le fichier SQL est injecté dans PostgreSQL via `psql`

[TIP]
====
La restauration écrase les données existantes. Faites une sauvegarde avant de restaurer.
====

=== Commandes manuelles (avancé)

[source,bash]
----
# Lister les fichiers de sauvegarde
docker compose exec api ls -lh /backups/pgdump/

# Restaurer manuellement
docker compose exec -T db psql -U postgres -d mydb < backup.sql

# Créer une sauvegarde manuelle via pg_dump
docker compose exec db pg_dump -U postgres -d mydb \
  --data-only --inserts --no-owner --no-acl \
  -f /tmp/backup_manuelle.sql

# Télécharger une sauvegarde
docker compose cp api:/backups/pgdump/full_20260427_123456.sql .
----

== Étape 5 : Logs

=== Triple stockage

Les logs API sont écrits simultanément dans 3 supports :

[cols="1,2,2", options="header"]
|===
| Stockage | Durée | Commande d'accès
| **RAM** (500 entrées) | Perdu au redémarrage | Interface admin
| **PostgreSQL** | 90 jours (purge auto) | Interface admin ou SQL
| **Fichier** (JSONL) | Illimitée | `docker compose exec api tail -f /var/log/api/api.log`
|==|

=== Consultation des logs fichier

[source,bash]
----
# En temps réel
docker compose logs -f api

# Dernières 50 lignes du fichier
docker compose exec api tail -50 /var/log/api/api.log

# Filtrer les erreurs
docker compose exec api grep ERROR /var/log/api/api.log

# Compter le nombre de logs
docker compose exec api wc -l /var/log/api/api.log
----

=== Purge des logs

[source,bash]
----
# Depuis l'interface : Sauvegardes → Logs API → "Vider"
# Ou en SQL directement :
docker compose exec db psql -U postgres -d mydb \
  -c "DELETE FROM api_logs;"
----

== Dépannage

=== Problème : les conteneurs ne démarrent pas

[source,bash]
----
# Voir les logs détaillés
docker compose logs

# Vérifier les permissions
sudo chown -R $USER:$USER /opt/Projet-Fil-Rouge-Dev

# Redémarrer Docker
sudo systemctl restart docker
docker compose up -d
----

=== Problème : "port already allocated"

[source,bash]
----
# Vérifier quel processus utilise le port
sudo lsof -i :3000
sudo lsof -i :8080
sudo lsof -i :5432

# Modifier les ports dans docker-compose.yml si nécessaire
# Remplacer "3000:3000" par "3001:3000"
----

=== Problème : erreur 429 (Too Many Requests)

Le rate limiting est atteint. Attendre 60 secondes que le quota se réinitialise.

[source,bash]
----
# Réinitialiser le rate limiting en redémarrant l'API
docker compose restart api
----

=== Problème : base de données corrompue

[source,bash]
----
# Réinitialiser complètement (⚠️ perd toutes les données)
docker compose down -v
docker compose up -d
----

=== Problème : les emails ne partent pas

[source,bash]
----
# Vérifier la clé Resend
docker compose exec web env | grep RESEND

# Tester l'envoi
curl -X POST http://localhost:3000/api/test/send-email \
  -H "Content-Type: application/json" \
  -d '{"type":"welcome", "email":"test@example.com", "firstName":"Test"}'

# Vérifier les logs Node.js
docker compose logs web | grep -i email
----

=== Annexe B : Sécurité — Rate Limiting

[cols="1,2,2", options="header"]
|===
| Couche | Limiteur | Limite
| Node.js | `strictLimiter` | 120 req/min sur `/admin/api`
| Node.js | `apiLimiter` | 100 req/min sur `/api`
| Node.js | `loginLimiter` | 5 req/15min sur login
| Go API | `RateLimitAPI` | 100 req/min (global)
| Go API | `RateLimitAdmin` | 600 req/min (admin)
| Frontend | Circuit breaker | Attente 60s après un 429
|===
